# PRD Workflow Skill v3 - MVP 중심 PRD 생성

이 스킬은 사용자의 아이디어를 **최소 범위의 MVP PRD**로 변환하고, **객관적 채점 루브릭**으로 검증하며, 최종 PRD를 **개발용 기술 명세**로 변환합니다.

---

## v2 → v3 핵심 변경사항

| 영역 | v2 | v3 |
|------|----|----|
| MVP 정의 | P0 + P1 허용 | **P0만 MVP** (하드 리밋) |
| 범위 제한 | 없음 | **기능 3개, API 5개, 화면 4개** |
| 채점 루브릭 | 완전성 30점 | 완전성 20점 + **간결성 20점** |
| 피드백 방향 | 추가 유도 | **축소/단순화 우선** |
| 스코프 질문 | 일반적 | **"하루 만에 만든다면?"** 포함 |

---

## 핵심 원칙

> **"작게 시작하라. 나중에 확장하라."**

MVP는 **핵심 가치 검증**이 목적입니다. 기능이 많으면 MVP가 아닙니다.

> **"줄이는 것이 추가하는 것보다 어렵다."**

기능을 추가하기 전에 "이것 없이 핵심 가치를 보여줄 수 있는가?"를 먼저 묻습니다.

> **"하드 리밋을 존중하라."**

MVP 범위 제한을 초과하면 채점에서 감점됩니다. 예외 없음.

## Phase 1: 아이디어 수집 + MVP 스코프 확정 ⭐

### 1.1 MVP 범위 확정 질문 (필수)

아이디어를 받은 후 **반드시** 아래 질문을 통해 범위를 좁힙니다:

| # | 질문 | 목적 |
|---|------|------|
| 1 | **딱 한 가지만 되면?** 이 앱의 핵심 가치를 보여주는 기능 1개는? | 핵심 기능 식별 |
| 2 | **3개로 제한하면?** 가장 중요한 기능 3개만 고르면? | P0 기능 확정 |
| 3 | **하루(8시간) 만에 만든다면?** 그 시간 안에 만들 수 있는 범위는? | 현실적 범위 설정 |
| 4 | **없어도 되는 것?** 처음부터 넣을 필요 없는 기능은? | Out of Scope 명확화 |
| 5 | **누가 쓰나?** 첫 번째 사용자 1명을 구체적으로 묘사하면? | 타겟 집중 |

### 1.2 MVP 하드 리밋 (Hard Limit)

**이 제한을 초과하면 채점에서 자동 감점됩니다.**

| 항목 | 최대값 | 초과 시 |
|------|--------|---------|
| 핵심 기능 (P0) | **3개** | -10점 (F1) |
| API 엔드포인트 | **5개** | -5점 (F2) |
| 화면 | **4개** | -5점 (F3) |
| Data Model 엔티티 | **4개** | -3점 (F4) |
| User Story | **5개** | Warning |

### 1.3 우선순위 정의 (명확화)

| 등급 | 의미 | MVP 포함 | 예시 |
|------|------|----------|------|
| **P0** | 이것 없으면 앱이 아님 | ✅ **필수** | 로그인, 핵심 기능 |
| P1 | 있으면 좋음 | ❌ v1.1로 연기 | 알림, 설정 |
| P2 | 나중에 | ❌ v2.0로 연기 | 공유, 통계 |

> **MVP = P0만**. P1은 "있으면 좋음"이지 "필수"가 아닙니다.

## Phase 2: PRD 초안 생성

### 2.1 PRD 필수 섹션 (간소화)

| # | 섹션 | 필수 여부 | 설명 |
|---|------|-----------|------|
| 1 | Executive Summary | ✅ | 3문장 이내 요약 |
| 2 | Problem Statement | ✅ | 해결할 문제 1개 |
| 3 | Goals & Success Metrics | ✅ | 목표 1-2개 + 지표 |
| 4 | User Persona | ✅ | **1개만** (핵심 사용자) |
| 5 | User Stories | ✅ | **3-5개** (P0만) |
| 6 | Functional Requirements | ✅ | **3개** (MVP 기능) |
| 7 | Non-Functional Requirements | 권장 | 성능, 보안 (간략히) |
| 8 | Technical Requirements | ✅ | 기술 스택 |
| 9 | UI/UX Requirements | ✅ | 화면 **4개 이내** |
| 10 | API Specifications | ✅ | **5개 이내** |
| 11 | Data Model | ✅ | **4개 이내** 엔티티 |

**v2 대비 변경:**
- User Persona: 2개 → **1개**
- User Stories: 5개 이상 → **3-5개**
- API: 제한 없음 → **5개 이내**

### 2.2 prd-architect 호출

```markdown
Task tool:
- subagent_type: "prd-architect"
- prompt: |
    ## 작업 요청: MVP PRD 초안 작성

    ### 참조 문서
    `.prd-workspace/scope.md`를 **반드시** 먼저 읽고 MVP 하드 리밋을 확인하세요.

    ### MVP 하드 리밋 (절대 초과 금지)
    - 핵심 기능: 3개
    - API: 5개
    - 화면: 4개
    - 엔티티: 4개

    ### PRD 작성 규칙

    1. **간결하게 작성** - 불필요한 설명 제거
    2. **P0만 포함** - P1, P2는 "Future Considerations"로 분리
    3. **User Persona 1개** - 핵심 사용자만
    4. **User Stories 3-5개** - P0 기능에 대응하는 것만

    ### User Story 형식
    ```
    US-001: As a [사용자], I want [기능], so that [가치].
    Acceptance Criteria:
    - [ ] 조건 1
    - [ ] 조건 2
    ```

    ### FR 형식
    ```
    FR-001: [기능명]
    - 설명: (1-2문장)
    - 입력: ...
    - 출력: ...
    - 관련 US: US-001
    ```

    ### 저장 위치
    `.prd-workspace/prd-v1.md`

- description: "MVP PRD 초안 작성"
```

## Phase 3: 루브릭 기반 검토 ⭐ 간결성 추가

### 3.1 채점 루브릭 (100점 만점)

> **v3 변경: 완전성 30→20점, 간결성 20점 신설**

#### 섹션 A: 완전성 (20점) ← 감소

| # | 체크 항목 | 배점 | 판단 기준 |
|---|-----------|------|-----------|
| A1 | Executive Summary 존재 | 2 | 있고 3문장 이내면 ✅ |
| A2 | Problem Statement 존재 | 2 | 문제가 명확하면 ✅ |
| A3 | Goals & Metrics 존재 | 2 | 목표+지표 있으면 ✅ |
| A4 | User Persona 존재 | 2 | 1개 이상이면 ✅ |
| A5 | User Stories 존재 | 2 | 3개 이상이면 ✅ |
| A6 | Functional Requirements 존재 | 2 | FR 있으면 ✅ |
| A7 | Technical Requirements 존재 | 2 | 기술 스택 명시면 ✅ |
| A8 | API Specifications 존재 | 2 | API 1개 이상이면 ✅ |
| A9 | Data Model 존재 | 2 | 엔티티 1개 이상이면 ✅ |
| A10 | UI/UX Requirements 존재 | 2 | 화면 설명 있으면 ✅ |

#### 섹션 B: 명확성 (20점)

| # | 체크 항목 | 배점 | 판단 기준 |
|---|-----------|------|-----------|
| B1 | US 형식 준수 | 4 | 모든 US가 "As a... I want... So that..." |
| B2 | US에 AC 있음 | 4 | 모든 US에 Acceptance Criteria |
| B3 | FR에 입력/출력 명시 | 4 | 모든 FR에 입출력 정의 |
| B4 | API에 Req/Res 명시 | 4 | 모든 API에 형식 정의 |
| B5 | Metrics 측정 가능 | 4 | 지표에 목표값+측정방법 |

#### 섹션 C: 일관성 (15점)

| # | 체크 항목 | 배점 | 판단 기준 |
|---|-----------|------|-----------|
| C1 | US-FR 연결 | 5 | 모든 FR이 US와 연결 |
| C2 | FR-API 연결 | 5 | 주요 FR에 대응 API 존재 |
| C3 | Scope 일치 | 5 | PRD가 scope.md 범위 내 |

#### 섹션 D: 실현 가능성 (15점)

| # | 체크 항목 | 배점 | 판단 기준 |
|---|-----------|------|-----------|
| D1 | 기술 스택 구체적 | 5 | 언어, 프레임워크 명시 |
| D2 | 외부 의존성 명시 | 5 | 필요한 외부 서비스 나열 |
| D3 | 복잡도 적절 | 5 | 8시간 내 개발 가능해 보임 |

#### 섹션 E: 개발 준비도 (10점)

| # | 체크 항목 | 배점 | 판단 기준 |
|---|-----------|------|-----------|
| E1 | API 정의 충분 | 4 | 필요한 API가 다 정의됨 |
| E2 | Data Model 충분 | 3 | 필요한 엔티티가 다 정의됨 |
| E3 | 에러 처리 명시 | 3 | API에 에러 코드 있음 |

#### 섹션 F: 간결성 (20점) ⭐ 신설

| # | 체크 항목 | 배점 | 판단 기준 |
|---|-----------|------|-----------|
| F1 | MVP 기능 3개 이하 | 10 | P0 기능 ≤ 3개면 ✅ (초과 시 -10) |
| F2 | API 5개 이하 | 5 | API ≤ 5개면 ✅ (초과 시 -5) |
| F3 | 화면 4개 이하 | 3 | 화면 ≤ 4개면 ✅ (초과 시 -3) |
