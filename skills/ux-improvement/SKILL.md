# UX Improvement Skill

UI/UX를 분석하고 개선하는 워크플로우입니다.

---

## 핵심 원칙

> **"기능은 그대로, 경험만 개선하라."**

비즈니스 로직 변경 없이 사용자 인터페이스와 경험만 개선합니다.

> **"사용자에게 선택권을 주어라."**

발견된 모든 이슈를 자동으로 고치지 않고, 우선순위와 함께 제시하여 선택하게 합니다.

> **"측정 가능한 개선을 하라."**

"더 좋아졌다"가 아닌 "접근성 점수 65→85" 같은 객관적 개선.

---

## 사용 시점

| 상황 | 적합 |
|------|------|
| "UI 좀 이쁘게 해줘" | ✅ |
| "다크모드 지원해줘" | ✅ |
| "모바일에서 안 이뻐" | ✅ |
| "접근성 개선해줘" | ✅ |
| "로딩이 느려" (기능 문제) | ⚠️ 일부만 |
| "새 기능 추가해줘" | ❌ (feature-extension) |

---

## 워크플로우 개요

```
Phase 1: 현재 UI 분석
    ↓
Phase 2: 문제점 보고 및 우선순위 제시
    ↓
Phase 3: 사용자 선택 (어떤 것 개선할지)
    ↓
Phase 4: 개선 계획 수립
    ↓
Phase 5: 개선 구현
    ↓
Phase 6: 검증 및 전후 비교
```

---

## Phase 1: 현재 UI 분석

### 목적

현재 UI의 상태를 객관적으로 분석합니다.

### 에이전트

`ux-auditor`

### MCP 도구 활용 (연결된 경우)

- **Figma MCP 연결 시**: Figma 파일에서 직접 화면별 디자인 분석, Variables/Styles 일관성 점검
- **Sketch MCP 연결 시**: Sketch 문서에서 화면 캡처, Shared Styles 분석
- **MCP 미연결 시**: 기존 코드 기반 분석 워크플로우로 진행 (fallback)

### 아이콘 사용 현황 점검 (Critical 이슈)

> **이모지를 UI 아이콘으로 사용하고 있으면 Critical 이슈로 분류한다.**

```markdown
## 아이콘 점검 항목

| # | 점검 항목 | 심각도 | 판단 기준 |
|---|----------|--------|-----------|
| 1 | 이모지를 UI 아이콘으로 사용 | Critical | 코드에서 이모지 문자 검색 |
| 2 | 아이콘 라이브러리 미사용 | Major | package.json에 lucide/heroicons 등 부재 |
| 3 | 아이콘 스타일 불일치 | Major | outlined + filled 혼용 |
| 4 | 아이콘 크기 불일치 | Minor | 동일 맥락에서 다른 크기 사용 |
```

### 작업 내용

```markdown
## 분석 영역 (7개 — 아이콘/디자인 품질 추가)

### 1. 접근성 (Accessibility)
- 색상 대비 (WCAG AA: 4.5:1)
- 키보드 접근성
- 스크린 리더 호환성
- ARIA 라벨
- 포커스 표시

### 2. 일관성 (Consistency)
- 버튼 스타일 통일
- 간격/여백 시스템
- 색상 팔레트
- 타이포그래피
- 아이콘 스타일

### 3. 사용성 (Usability)
- 클릭 영역 크기 (최소 44x44px)
- 로딩 상태 피드백
- 에러 상태 표시
- 빈 상태 (empty state)
- 액션 피드백

### 4. 반응형 (Responsive)
- 모바일 레이아웃
- 태블릿 레이아웃
- 터치 타겟 크기
- 콘텐츠 가독성

### 5. 다크 모드 (Dark Mode)
- 전체 컴포넌트 지원 여부
- 색상 대비 (다크모드)
- 이미지/아이콘 대응
- 전환 애니메이션

### 6. 성능 UX (Performance UX)
- 초기 로딩 경험
- 인터랙션 응답성
- 레이아웃 시프트
- 스켈레톤 UI

### 7. 디자인 품질 (Design Quality) ⭐ 신규
- 아이콘: 이모지 사용 여부 (Critical)
- 아이콘: 라이브러리 일관성
- 색상: 시맨틱 변수 사용 여부 (Tailwind 기본 색상 직접 사용 = Major)
- 폰트: 앱 성격에 맞는 폰트 사용 여부
- 시각적 위계: 간격/크기의 의도적 차등
- 전체 톤: 앱 목적에 맞는 시각적 방향성
```

### 결과물

- `.agent-results/ux-auditor/YYYY-MM-DD-ux-audit.md`

### 출력 형식

```markdown
## UX 감사 결과 요약

### 점수

| 영역 | 점수 | 등급 |
|------|------|------|
| 접근성 | 65/100 | C |
| 일관성 | 80/100 | B |
| 사용성 | 70/100 | C |
| 반응형 | 85/100 | B |
| 다크모드 | 40/100 | F |
| 성능 UX | 75/100 | C |
| **종합** | **69/100** | **C** |

### 이슈 요약

| 심각도 | 개수 |
|--------|------|
| Critical | 2 |
| Major | 8 |
| Minor | 12 |
| Enhancement | 5 |
```

## Phase 2: 문제점 보고 및 우선순위 제시

### 목적

발견된 이슈를 사용자에게 보고하고 선택할 수 있게 합니다.

### 작업 내용

```markdown
## 개선 제안 목록

### 🔴 Critical (즉시 개선 권장)

| # | 이슈 | 영향 | 예상 소요 |
|---|------|------|-----------|
| 1 | 버튼 키보드 접근 불가 | 키보드 사용자 차단 | 30분 |
| 2 | 색상 대비 부족 (로그인 폼) | 저시력 사용자 | 15분 |

### 🟠 Major (개선 권장)

| # | 이슈 | 영향 | 예상 소요 |
|---|------|------|-----------|
| 3 | 로딩 상태 없음 | 사용자 혼란 | 1시간 |
| 4 | 다크모드 미지원 (50%) | 야간 사용자 | 2시간 |
| 5 | 모바일 테이블 잘림 | 모바일 사용자 | 1시간 |

### 🟡 Minor (선택적 개선)

| # | 이슈 | 영향 | 예상 소요 |
|---|------|------|-----------|
| 6 | 버튼 스타일 불일치 | 시각적 일관성 | 1시간 |
| 7 | 간격 불규칙 | 시각적 품질 | 30분 |

### 🔵 Enhancement (추가 개선)

| # | 제안 | 효과 | 예상 소요 |
|---|------|------|-----------|
| 8 | 스켈레톤 UI 추가 | 체감 속도 향상 | 2시간 |
| 9 | 마이크로 애니메이션 | 인터랙션 품질 | 1시간 |
```

### 사용자 선택 요청

```markdown
## 어떤 항목을 개선할까요?

**추천 패키지**:
- A. Critical만 (약 45분) - 최소 필수
- B. Critical + Major (약 5시간) - 권장
- C. 전체 (약 10시간) - 완전 개선

**또는 번호로 선택**: "1, 3, 4, 6번 해줘"
```

## Phase 3: 사용자 선택

### 목적

사용자가 개선할 항목을 선택합니다.

### 입력 예시

```
"Critical이랑 다크모드만 해줘"
"B 패키지로 해줘"
"1, 3, 4번만"
```

### 결과물

- 선택된 개선 항목 목록
- 예상 총 소요 시간

## Phase 4: 개선 계획 수립

### 목적

선택된 항목들의 구현 순서와 방법을 계획합니다.

### 작업 내용

```markdown
## 개선 계획

### 선택된 항목
- #1: 버튼 키보드 접근
- #3: 로딩 상태 추가
- #4: 다크모드 지원

### 구현 순서

| 순서 | 항목 | 영향 파일 | 선행 조건 |
|------|------|-----------|-----------|
| 1 | #1 키보드 접근 | IconButton.tsx (1개) | 없음 |
| 2 | #4 다크모드 | 12개 컴포넌트 | 없음 |
| 3 | #3 로딩 상태 | 5개 컴포넌트 | 없음 |

### 수정할 파일

| 파일 | 개선 항목 | 변경 내용 |
|------|-----------|-----------|
| components/ui/IconButton.tsx | #1 | button 태그로 변경 |
| components/ui/Badge.tsx | #4 | dark: 클래스 추가 |
| components/features/StockList.tsx | #3, #4 | 로딩 상태 + 다크모드 |
| ... | ... | ... |
```

### 결과물

- `.agent-results/ux-improvement/YYYY-MM-DD-improvement-plan.md`

## Phase 5: 개선 구현

### 목적

계획대로 UI를 개선합니다.

### 에이전트

`frontend-developer`

### 작업 규칙

```markdown
## 구현 규칙

### 1. 기능 변경 금지
- 비즈니스 로직 수정 ❌
- 상태 구조 변경 ❌
- API 호출 변경 ❌
- 스타일/UX만 수정 ✅

### 2. 기존 패턴 유지
- 컴포넌트 구조 유지
- 파일 위치 유지
- 네이밍 규칙 유지

### 3. 점진적 개선
- 한 파일씩 수정
- 수정 후 빌드 확인
- 기존 기능 동작 확인

### 4. 코드 예시 활용
- ux-auditor가 제시한 개선 코드 참조
- 필요시 프로젝트 컨벤션에 맞게 조정
```

### 개선 전/후 코드 예시

```markdown
## #1 키보드 접근 개선

### Before
```tsx
<div onClick={handleClick} className="cursor-pointer">
  <Icon />
</div>
```

### After
```tsx
<button
  onClick={handleClick}
  className="cursor-pointer focus:ring-2 focus:ring-blue-500 focus:outline-none"
  aria-label="아이콘 버튼"
>
  <Icon />
</button>
```
```

### 결과물

- 수정된 코드 파일들
- `.agent-results/frontend/YYYY-MM-DD-ux-improvement.md`

## Phase 6: 검증 및 전후 비교

### 목적

개선 결과를 검증하고 전후 비교를 제공합니다.

### 에이전트

`ux-auditor`, `test-runner`

### 작업 내용

```markdown
## 검증 항목

### 1. 빌드 검증
- npm run build 성공 여부

### 2. 기존 기능 검증
- 테스트 전체 실행
- 수동 기능 확인 (주요 흐름)

### 3. 개선 효과 측정
- 각 항목별 Before/After
- 점수 변화
```

### 결과물

```markdown
## 개선 결과 리포트

**개선일**: YYYY-MM-DD
**개선 항목**: 3개

---

## 전후 비교

### 점수 변화

| 영역 | Before | After | 변화 |
|------|--------|-------|------|
| 접근성 | 65 | 85 | +20 ⬆️ |
| 일관성 | 80 | 80 | 0 |
| 사용성 | 70 | 82 | +12 ⬆️ |
| 다크모드 | 40 | 95 | +55 ⬆️ |
| **종합** | **69** | **85** | **+16 ⬆️** |

### 항목별 결과

| # | 항목 | 결과 | 비고 |
|---|------|------|------|
| 1 | 키보드 접근 | ✅ 완료 | IconButton 수정 |
| 3 | 로딩 상태 | ✅ 완료 | 5개 컴포넌트 |
| 4 | 다크모드 | ✅ 완료 | 12개 컴포넌트 |

### 시각적 비교

| 화면 | Before | After |
|------|--------|-------|
| 메인 페이지 | 다크모드 미지원 | 다크모드 지원 ✅ |
| 로딩 상태 | 빈 화면 | 스켈레톤 UI ✅ |

---

## 검증 결과

| 항목 | 결과 |
|------|------|
| 빌드 | ✅ 성공 |
| 테스트 | ✅ 전체 통과 |
| 기존 기능 | ✅ 정상 동작 |

---

## 남은 개선 사항

| # | 항목 | 심각도 | 비고 |
|---|------|--------|------|
| 6 | 버튼 스타일 통일 | Minor | 다음에 진행 가능 |
| 7 | 간격 정리 | Minor | 다음에 진행 가능 |

## 전체 흐름 예시

```
사용자: "UI 좀 이쁘게 해줘"
    ↓
Phase 1: ux-auditor
    → 분석 결과: 종합 69점, Critical 2개, Major 8개
    ↓
Phase 2: 보고
    → "27개 개선점 발견. 어떤 것 할까요?"
    → A/B/C 패키지 또는 번호 선택
    ↓
Phase 3: 사용자 선택
    → "B 패키지로 해줘"
    ↓
Phase 4: 계획 수립
    → Critical 2개 + Major 8개 = 10개 항목
    → 영향 파일 20개, 예상 5시간
    ↓
Phase 5: frontend-developer
    → 순서대로 구현
    → 각 수정 후 빌드 확인
    ↓
Phase 6: 검증
    → 점수: 69 → 85 (+16)
    → 테스트 전체 통과
    → 전후 비교 리포트
    ↓
완료: "UX 개선 완료! 점수 69→85 (+16점)"
```

## 에스컬레이션 정책

| 조건 | 동작 |
|------|------|
| 분석 중 기능 버그 발견 | 보고만, UX 개선만 진행 |
| 개선 중 빌드 실패 | 해당 변경 롤백, 다른 항목 계속 |
| 기존 테스트 실패 | 즉시 중단, 원인 분석 |
| 예상보다 영향 범위 큼 | 사용자에게 보고, 계속 여부 확인 |
